- 2026-07-11: T11's `AgentState` is a plain `TypedDict` (not a Pydantic model) so LangGraph's default per-key "overwrite, else keep checkpointed value" merge semantics apply with no custom reducers needed. `ui` is stored as a plain dict inside state and only validated against `app.models.ui.UICommand` at the API boundary (main.py), since the checkpointer just needs to serialize/deserialize state, not enforce the UI contract.
- 2026-07-11: The graph is one-node-per-turn by design: `START` conditionally routes to whichever node matches the checkpointed `stage` (`route_by_stage`), and every node's only outgoing edge is `END`. This matches T11's instruction to "run one graph step" per `/api/turn` call — the *next* stage a node writes into state is what the *next* turn's dispatch will route to, not a multi-node walk within one call.
- 2026-07-11: Guard rules (Spec S9.1 "teach unreachable without a profile") are enforced as a single layer inside `teach.py` itself (see `app/agent/guards.py`), not duplicated at the router level — an earlier router-level redirect attempt made the guard's effect invisible in tests, since `confirm_profile` (the redirect target) itself advances back to `teach`, making the final `stage` identical whether or not the guard fired. Checking at the node level makes "teach was blocked" directly observable: `result["stage"] == "confirm_profile"`, not `"teach"`.
- 2026-07-11: The reteach-cap guard (`can_start_reteach`, max 2 rounds per concept per Spec S9.1) is implemented and unit-tested as a pure function now, but not yet wired into `reteach.py`'s node logic -- there's no real per-concept grading to trigger reteach until T19/T20, so the skeleton's `viva` node always advances to `wrapup` rather than ever routing to `reteach`.

- 2026-07-11: First live browser testing surfaced four real, previously-invisible bugs (found using an automated Playwright session with a fake-microphone Chromium instance, since manual back-and-forth over chat console-log pasting was too slow to iterate on):
  1. **STT rejected all real browser audio.** Sarvam's content-type allow-list rejects `audio/webm;codecs=opus` (what `MediaRecorder` blobs actually report) but accepts bare `audio/webm`. `stt.py::transcribe()` now strips any `;codecs=...` parameter before sending. Regression-tested.
  2. **The frontend never called `POST /api/session`.** Since T08 (before T09's DB / T11's graph existed), the frontend generated its own `session_id` via `crypto.randomUUID()` and called `/api/turn` directly. That was harmless against the old echo-stub, but T11's real graph requires a checkpoint to already exist for that thread_id -- calling `/api/turn` cold now throws `KeyError: 'stage'` inside `route_by_stage`, a 500 that (see next bug) looked like a CORS failure in the browser. Fixed: `App.tsx` now calls `postSession()` on mount and blocks the talk button (shows "connecting...") until a real `session_id` comes back.
  3. **A 500 from our own `Exception` handler carried no CORS headers**, so the browser reported "blocked by CORS policy" instead of showing the real 500 body -- actively hiding the bug above during debugging. Cause: Starlette's `ServerErrorMiddleware` (which invokes handlers registered via `add_exception_handler(Exception, ...)`) sits *outside* `CORSMiddleware`, so CORS headers never get added to that path. `errors.py`'s two handlers now add `Access-Control-Allow-Origin` manually by checking the request's `Origin` header against configured `CORS_ORIGINS`.
  4. **Push-to-talk race condition.** `getUserMedia()` is async and can take longer than a fast tap-and-release; releasing before it resolves left `isRecording` false when `pointerup` fired, so `handlePressEnd` did nothing and the mic silently stayed "listening" forever with nothing left to ever call `stop()`. Fixed with a `startingRef`/`releaseRequestedRef` pair: a release that arrives mid-start is remembered and finishes the turn the moment `start()` resolves, instead of being dropped.
- 2026-07-11: Reply audio playback moved from `new Audio(dataUri).play()` to the Web Audio API (`AudioContext` + `decodeAudioData`), with the context resumed synchronously at the start of the pointerdown handler (before any `await`). Turn round trips (STT+agent+TTS) commonly take 2-6s -- long enough that a fresh `<audio>` element's `.play()` call made *after* that delay could get silently rejected by the browser's autoplay policy, since transient user-activation from the original gesture had already expired. Resuming the context during the gesture keeps it unlocked for the rest of the session.

- 2026-07-11: **A live scripted run in Hindi surfaced a real content-alignment bug**, invisible to the mocked-LLM unit tests since those don't exercise real model behavior: `discover.step0`'s "ask village/work" call was passing her actual prior transcript (which belonged to *greet's* consent question) into the LLM. A real model is smart enough to react to whatever it's shown, so it improvised straight past the intended question, jumping ahead to ask about her skill interest in the same reply -- desyncing the `stage_step` counter from what was *actually* asked. The knock-on effect: `discover.step1`'s `VillageWorkExtraction` then ran against the string `"tailoring"` (the answer to a question that hadn't structurally been asked yet), producing garbage. Fixed by not forwarding `transcript` on the two other purely-content-agnostic "step0 ask" calls (`discover.step0`, `confirm_profile.step0`) -- confirmed via a second live run that the corrected flow asks each question exactly once, in order, with extraction landing on the right turn's content. Cost: an extra "okay"-style filler turn at each of these two stage transitions, since the transition reply itself doesn't ask the next stage's first question. A future refinement could combine ack + first-question into one reply to remove it; not done here given time.
- 2026-07-11: A second live-run bug in the same family: `assess`'s per-question prompt never told the model what was already known about her (name/village/interest) or showed it the conversation so far, so occasionally (temperature=0.7, not every run) it would re-ask background questions instead of a skill-story question. Fixed by passing her known profile facts plus a rendered transcript of `state["history"]` into every question's instruction, not just the final extraction call. This *reduced* but did not fully eliminate drift -- a Gujarati live run still saw one re-asked question after the fix. That residual is LLM stochasticity, not a code bug: same code, clean run in Hindi, one drift in Gujarati. Documented as a known limitation for future prompt iteration (few-shot examples would likely tighten it further); the conversation still completes correctly and the saved profile is accurate either way.
