- 2026-07-11: First live browser testing surfaced four real, previously-invisible bugs (found using an automated Playwright session with a fake-microphone Chromium instance, since manual back-and-forth over chat console-log pasting was too slow to iterate on):
  1. **STT rejected all real browser audio.** Sarvam's content-type allow-list rejects `audio/webm;codecs=opus` (what `MediaRecorder` blobs actually report) but accepts bare `audio/webm`. `stt.py::transcribe()` now strips any `;codecs=...` parameter before sending. Regression-tested.
  2. **The frontend never called `POST /api/session`.** Since T08 (before T09's DB / T11's graph existed), the frontend generated its own `session_id` via `crypto.randomUUID()` and called `/api/turn` directly. That was harmless against the old echo-stub, but T11's real graph requires a checkpoint to already exist for that thread_id -- calling `/api/turn` cold now throws `KeyError: 'stage'` inside `route_by_stage`, a 500 that (see next bug) looked like a CORS failure in the browser. Fixed: `App.tsx` now calls `postSession()` on mount and blocks the talk button (shows "connecting...") until a real `session_id` comes back.
  3. **A 500 from our own `Exception` handler carried no CORS headers**, so the browser reported "blocked by CORS policy" instead of showing the real 500 body -- actively hiding the bug above during debugging. Cause: Starlette's `ServerErrorMiddleware` (which invokes handlers registered via `add_exception_handler(Exception, ...)`) sits *outside* `CORSMiddleware`, so CORS headers never get added to that path. `errors.py`'s two handlers now add `Access-Control-Allow-Origin` manually by checking the request's `Origin` header against configured `CORS_ORIGINS`.
  4. **Push-to-talk race condition.** `getUserMedia()` is async and can take longer than a fast tap-and-release; releasing before it resolves left `isRecording` false when `pointerup` fired, so `handlePressEnd` did nothing and the mic silently stayed "listening" forever with nothing left to ever call `stop()`. Fixed with a `startingRef`/`releaseRequestedRef` pair: a release that arrives mid-start is remembered and finishes the turn the moment `start()` resolves, instead of being dropped.
- 2026-07-11: Reply audio playback moved from `new Audio(dataUri).play()` to the Web Audio API (`AudioContext` + `decodeAudioData`), with the context resumed synchronously at the start of the pointerdown handler (before any `await`). Turn round trips (STT+agent+TTS) commonly take 2-6s -- long enough that a fresh `<audio>` element's `.play()` call made *after* that delay could get silently rejected by the browser's autoplay policy, since transient user-activation from the original gesture had already expired. Resuming the context during the gesture keeps it unlocked for the rest of the session.

- 2026-07-11: React StrictMode's dev-only double-invoke of effects was firing `POST /api/session` twice per page load (each one a real live TTS call). Guarded with a `useRef` flag checked at the top of the effect -- deliberately *not* the usual `cancelled` closure-flag pattern, because StrictMode's synthetic first cleanup would set that flag on the one call that actually runs (the ref guard suppresses the second invocation entirely), silently discarding its result.
